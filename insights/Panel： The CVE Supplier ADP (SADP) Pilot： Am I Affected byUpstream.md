# Panel： The CVE Supplier ADP (SADP) Pilot： Am I Affected byUpstream
## Executive Summary
This panel, featuring experts from MITRE, CISA, Microsoft, Red Hat, and Qualys, discusses the Supplier Authorized Data Provider (SADP) pilot. The initiative aims to solve the "dependency gap" in vulnerability management by allowing downstream software suppliers to add their own exploitability assessments (VEX statements) directly to an upstream CVE record. By using ADP containers within the CVE JSON schema, suppliers can provide authoritative, machine-readable data on whether their products are affected, not affected, or mitigated. The discussion covers the technical challenges of scaling this model, the benefits for scanning vendors, and the balance between centralized and distributed vulnerability data.

## Key Points
- **The Dependency Problem:** Most software is built from other software. When an upstream library (like curl or OpenSSL) has a CVE, downstream suppliers often lack a standard way to tell customers if their specific implementation is affected.
- **SADP Mechanics:** The pilot uses "ADP (Authorized Data Provider) containers" within CVE records. This allows a supplier (e.g., Red Hat or Siemens) to add a vendor-specific section to a record they didn't originally assign.
- **VEX at Scale:** SADP is essentially "VEX for CVE." It allows vendors to publish "Not Affected" statements, which is critical for reducing the noise generated by security scanners that flag all instances of a library.
- **Upstream vs. Downstream:** Upstream maintainers (like kernel.org) often lack the context of how their code is used in a specific product (e.g., an Android phone vs. a data center). SADP allows downstream suppliers to fill this metadata gap.
- **Scaling Challenges:** If every supplier added a container for every dependency, CVE records could grow to unmanageable sizes (terabytes of data across the ecosystem). The program is exploring "robust external references" (pointers) as an alternative.
- **Trust and Provenance:** Each ADP container is owned by the organization that provided it. This allows consumers to trust the information based on the supplier's reputation (e.g., "Microsoft says Edge is not affected by this Chrome CVE").
- **Efficiency for Scanners:** Scanning vendors currently scrape hundreds of individual vendor feeds. SADP offers a "one-stop shop" for impact data, potentially reducing detection release times.
- **Infrastructure:** The pilot uses existing CVE Services and the 5.1 schema, requiring no major new technical breakthroughs but significant architectural decisions regarding data volume.
- **Early Success:** Participants like Siemens and Microsoft have already begun publishing production pilot data, proving the concept is technically viable.
- **Identity vs. Applicability:** The pilot highlights the need for better software identity (PURLs/CPEs) to ensure that the correct downstream products are linked to the correct upstream vulnerabilities.

## Takeaways
- **Check for ADP Badges:** When triaging a CVE on cve.org, look for the "ADP" badge. It may contain a "Not Affected" statement from your software supplier that overrides a generic high-severity score.
- **Prepare for VEX:** Suppliers should begin structuring their internal vulnerability data into VEX-compliant formats (CSAF, CycloneDX) to participate in future SADP-like ecosystems.
- **Standardize on PURL:** To make SADP work, both upstream and downstream must use consistent package identifiers (PURLs) to ensure accurate matching.
- **Demand Downstream Context:** Customers should ask their vendors to provide SADP or VEX data to avoid wasting time on "unexploitable" vulnerabilities flagged by SCA tools.
- **Support Federation:** Centralizing all vulnerability data is likely unsustainable. The community should support federated models where CVE records point to trusted vendor repositories.
